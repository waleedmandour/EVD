v1.5.1: Critical BLE/OBD-II fixes + real diagnostic scan

## What's Fixed

### ble-service.ts — Critical race condition + protocol parsing
- **Command queue (race condition fix)**: Concurrent `sendCommand` calls previously
  overwrote the single `responseResolve` callback, causing silent data corruption
  and unresolved promises during polling + UI-driven requests. Replaced with a
  proper FIFO command queue that serializes commands — concurrent callers now
  wait their turn and always receive their own response. The existing
  `commandQueue` / `isProcessingQueue` fields were declared but never used;
  they now drive the queue.
- **VIN multi-frame parsing**: Frame counter bytes (01, 02, 03) in multi-frame
  Mode 09 PID 02 responses were decoded as ASCII '1'/'2'/'3', then survived
  the VIN regex filter (digits are valid VIN chars) and corrupted the result.
  Now each `49 02 NN` frame header is detected and its 2-hex counter is
  skipped before decoding data bytes.
- **Disconnect cleanup**: Pending `sendCommand` promises now get rejected via
  `flushQueue()` on disconnect, so callers don't hang for the full timeout.
  Also clears `responseBuffer` and `responseResolve`.
- **Removed dead code**: `pollingInterval` field was cleared in `stopPolling`
  but never assigned anywhere (polling uses recursive `setTimeout` via
  `pollingTimeout`). Removed.
- **ATST64 comment fix**: 0x64 = 100, and 100 × 4 ms = **400 ms** per byte
  (not 100 ms as the comment claimed). The setting itself was already correct.

### store.ts — VIN parsing parity
- Applied the same multi-frame VIN fix to `parseOBDResponse` so streamed
  responses (via `bleService.startPolling`) are decoded identically to the
  one-shot read in `bleService.readVIN()`.

### DiagnosticsView.tsx — Real diagnostic scan (was fake)
- **`handleScan` was a no-op**: It only animated a progress bar 0→100 over
  2 seconds and never actually queried the vehicle. For BLE/WiFi connections
  it now issues Mode 03 (stored), Mode 07 (pending), and Mode 0A (permanent)
  DTC reads via `bleService.readDTCs()`, then merges the results with
  descriptions from `dtc-codes.ts`. Demo mode still uses the progress-bar
  fallback.
- **`handleClearCodes` only cleared local state**: It now also sends Mode 04
  to the vehicle ECU via `bleService.clearDTCs()`, so codes are actually
  erased from the vehicle (not just hidden in the UI).
- Added a `scanError` banner so adapter errors surface in the UI instead of
  being silently swallowed.
- Severity mapping: `DTCCode.severity` ('INFO'|'WARNING'|'CRITICAL') is
  properly mapped to `DTCEvent.severity` ('low'|'medium'|'high'|'critical').

### Build & version
- Bumped version to **1.5.1** in `package.json` and `android/app/build.gradle`
  (versionCode 8) to match the v1.5.1 README bump that was already on `main`.
- `build-apk.sh`: APK output filename now derives from `versionName` in
  `build.gradle` (was hardcoded to `evdx-v1.0.0-release.apk` regardless of
  the actual version).
- `android/app/build.gradle`: Release builds now use a proper release
  keystore (`evdx.keystore`, generated by `build-apk.sh`) when present,
  with safe fallback to debug signing. Previously, the generated keystore
  sat unused and release builds were always signed with the debug key.
- `tsconfig.json`: Excluded the unrelated `examples/`, `skills/`,
  `android/`, `android-template/`, and `mini-services/` template folders
  from type checking — they're not part of the EVDx app and were producing
  spurious TS errors.
- `.gitignore`: Added `*.keystore`, `*.jks`, `keystore.properties` so
  private signing keys never get committed.

## Note on hardware simulation

The task brief asked for a simulation to verify the APK detects OBD-II
devices and reads diagnostic data accurately. Without physical ELM327
hardware or an Android emulator with ELM327 simulation, true end-to-end
runtime testing against a live vehicle is not possible in this environment.
The included Node-based protocol simulation test exercises the exact byte-
level parsing logic that the production app uses, and is the closest
equivalent to a hardware simulation that can run in CI. For on-device
verification, the APK in this release can be sideloaded and tested against
a real OBD-II adapter — the BLE service changes are designed to be
backwards-compatible with all 10 supported adapter profiles.
